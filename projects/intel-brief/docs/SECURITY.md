# Intel Brief — Security

## API Key Handling

The Anthropic API key can be supplied two ways:

1. **Server-side** (`ANTHROPIC_API_KEY` env var): Never exposed to the browser.
   Used when you control the deployment and don't want users to supply their own key.

2. **Client-side** (user enters in browser): Stored in `localStorage` under `anthropic_api_key`.
   Sent as a JSON body field in the POST request to `/api/generate`.
   Never sent as a URL parameter or header that would appear in server logs.

**Risk**: `localStorage` is accessible to any JS running on the same origin.
For V1 (single-user local deploy or trusted users) this is acceptable.
For a multi-tenant SaaS, use server-side keys with per-user billing via the Anthropic API.

## Data Sent to Claude

The synthesis prompt includes:
- Scraped text content from source URLs (public web content)
- The previous briefing text (generated by Claude, stored by us)
- The topic name

It does **not** include:
- User PII
- API keys
- Anything from `profiles.json` beyond topic name and briefing history

The scraped content is truncated to 8000 chars per source. There is no mechanism
for source content to contain secrets because sources are public URLs.

## Source URL Trust

Users add source URLs. The scraper (`lib/scraper.ts`) fetches these with:
- 10 second timeout (prevents slow loris attacks)
- No redirect limit (TODO: cap at 3 redirects)
- No SSRF protection (V1 limitation: a user could add an internal URL)

**V1 risk**: If deployed internally, a user could add `http://169.254.169.254/latest/meta-data/`
(AWS metadata endpoint). The scraped content would be sent to Claude, not stored.
For internal deployments behind auth, this risk is low. For public SaaS: add an
allowlist of domains or a SSRF-safe HTTP client.

## Data Storage

`data/profiles.json` stores:
- Topic names, source URLs, source labels
- Full briefing text and citations
- No user credentials, no personal data beyond what the user explicitly enters

File permissions: writable only by the Node.js process. In a container, run as
non-root with restricted filesystem access.

## Authentication

V1 has no authentication. Assumes single-user local deployment or a trusted
team behind a VPN/auth proxy (e.g., Cloudflare Access, Tailscale Funnel).

For public SaaS V2: add NextAuth.js with GitHub/Google OAuth. Scope all
`profiles.json` reads to the authenticated user ID.

## Threat Model (V1 scope)

| Threat | Risk | Mitigation |
|--------|------|-----------|
| API key theft via XSS | Medium | CSP headers, no third-party scripts |
| SSRF via source URL | Low (no internal deploy) | Timeout, future: domain allowlist |
| Briefing data exposure | Low (local deploy) | Auth proxy in front (V2: NextAuth) |
| Prompt injection via source content | Low | Claude instruction-following is robust; citations require real URLs |
| Data loss (profiles.json) | Medium | Backup `data/` directory; V2: database with backups |
